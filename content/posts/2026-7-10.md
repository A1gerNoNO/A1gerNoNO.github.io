在真实公司里，研发不是凭空写功能，而是先接收产品 PRD，参与需求评审，理解业务背景、用户流程、交互效果和外部依赖，然后再做技术设计和开发。

这个项目的业务背景是：

```
公司想做一个自有在线商城
用于售卖电脑、键盘等产品
通过厂家直售减少中间渠道
给消费者更直接的购买方式和价格优势
```

系统整体目标是做一个小型支付商城，用户可以：

```
查看商品
扫码登录
下单购买
跳转支付宝支付
支付成功后完成订单流程
```

这章里的交互流程主要包括四部分：

```
商城首页
商城服务
微信公众号
支付宝支付
```

商城首页负责：

```
展示商品
触发扫码登录
保存登录 cookie
跳转支付宝收银台
```

商城服务负责：

```
获取微信登录凭证
查询商品
创建订单
封装支付宝支付
处理支付回调
发送登录通知模板
```

微信公众号用于：

```
扫码登录
降低用户注册登录成本
完成用户身份识别
```

支付宝用于：

```
创建支付交易
跳转收银台
完成真实支付流程
支付后回调商城服务
```

本章还提到研发需要提前准备三个外部平台：

```
微信公众测试平台：用于测试公众号扫码登录
支付宝沙箱：用于测试支付流程
natapp 内网穿透：用于本地开发时接收微信/支付宝回调
```

## 关于沙箱

** 沙箱环境是什么？**

沙箱环境就是一个“假的、隔离的测试环境”。
它的目的不是处理真实业务，而是让你安全地练习。
比如支付宝沙箱：

```text
你创建的订单是假的
用户付款是假的
钱不会真的转走
但接口流程和真实支付很像
```

它的价值是：

```text
不影响真实用户
不产生真实资金交易
可以反复测试
可以模拟支付成功、失败、回调等场景
```

你可以把它理解成：

```text
正式环境 = 真战场
沙箱环境 = 训练场
```

**4. Codex 的沙箱设计是什么？**

Codex 的沙箱和支付宝/微信沙箱不是一回事。

支付宝沙箱是 **业务测试环境**。  
Codex 沙箱是 **代码执行安全环境**。

Codex 沙箱的目的是保护你的电脑和项目，限制我能做什么。

比如你当前这个环境里，Codex 大概受到这些限制：

```text
可以读取允许范围内的文件
只能写入指定工作目录
不能随便改系统目录
网络访问受限制
危险操作需要用户批准
执行某些外部命令需要提权确认
```

所以 Codex 沙箱更像：

```text
我可以帮你写代码、读文件、运行命令
但不能随便碰你电脑里的所有东西
也不能随便联网或执行高风险操作

```

## 实际场景
在公司中实际做项目，会先有 产品拉一个评审会，对本地需求的 PRD 进行评审。这个过程会有产品、研发（前后端）、测试，一起参与评审。一般在 30分钟 ~ 2小时的会议。评审完成后，研发会对需求进行分工，较大型的项目，会有几个研发同时介入。之后每个研发进行设计阶段，设计会包括新项目的工程框架、技术栈选择、需求的模型设计（DDD）、功能的流程（MVC）实现、要实现和改动点说明等。
本节我们就按照这样的一个过程进行需求的研发设计，主要包括；用户用例图、四色建模（领域拆分）、UML流程图、改动点。有这些东西就可以指导研发开发了。
