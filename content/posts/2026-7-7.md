## 建模目的
用一套方法把业务流程、业务对象、业务边界梳理清楚，再知道后面的工程设计与代码开发

## 怎么建模
首先清楚：
```
系统有哪些业务流程
有哪些核心对象
有哪些领域边界
哪些地方要调用外部系统
哪些功能可以聚合在一起
后续代码应该怎么分包、怎么设计
```



## 什么是四色建模
```蓝色：决策命令：用户发起的动作
黄色：领域事件：事情已经完成之后的结果
红色：业务流程：从命令到事件之间，中间发生了什么处理
棕色：领域对象：业务过程中用到的重要对象
粉色：外部系统：当前系统需要调用的外部能力
绿色：只读模型：只查询数据、不修改模型的数据
```

## 建模思路
```从用户行为出发
找到用户发起的命令
分析命令会产生哪些领域事件
识别中间的业务流程和领域对象
最后划分领域边界
建模步骤可以总结为 5 步
分析产品诉求
先读懂需求，知道系统到底要做什么。
```

```
画用例图
从用户角度看系统有哪些功能。

寻找领域事件
找出系统中重要的业务结果，比如“抽奖完成”“奖品发放完成”。

识别领域角色和对象
分析是谁触发事件，需要哪些对象，中间经过什么流程。

划分领域边界
根据事件、流程和对象，把系统拆成不同领域，比如活动领域、账户领域、抽奖策略领域、奖品领域、库存领域。
```

## 一句话总结
DDD 建模要先从用户行为和业务事件出发，用四色建模找出命令、事件、流程、对象和外部系统，再据此划分领域边界，为后续代码设计打基础。
